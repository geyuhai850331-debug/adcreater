## Problem Statement

Agency account managers serving cross-border e-commerce clients spend ~70% of their time on integration labor — copy-pasting between ChatGPT and Midjourney, renaming files, resizing images, reformatting copy for different platforms. They manage 5-15 client brands simultaneously, each needing localized content for 2-4 channels. The existing AI tools are built for solo sellers, not agencies. The result: thin margins, slow turnaround, and account managers burning nights on file management instead of creative strategy.

## Demand Evidence

Paying users are asking for this — agency account managers who have offered to pay before the product exists. The specific pain is quantified: ~70% of a workday lost to integration labor. A tool that cuts this to ~10% is a 6x productivity multiplier for the agency's most expensive unit.

## Status Quo

Piecemeal AI tools: ChatGPT for copywriting, Midjourney/FLUX for images, manual Photoshop for assembly. Each listing requires 6-8 round trips across tools. Assets are organized in folders named by convention (`ClientX_AmazonJP_sku123_v3_final.png`). One mistake in file naming cascades into publishing errors. Parallel tools exist in the Chinese market (堆友AI, 美图设计室, 稿定设计) but none target the agency multi-client workflow.

## Target User & Narrowest Wedge

**Primary user**: Agency account manager — handles 5-15 client brands, produces 30-100 listings per week across 2-4 channels (Amazon JP, Amazon US, Shopee, social media). Promoted for client retention and throughput. Fired for errors and missed deadlines.

**Wedge**: A single-page Agency Command Center. Select client brand → input product info → generate copy + images in one pass → download a platform-ready ZIP with named files and manifest. One click per listing, not 8 round trips across 4 tools.

## Constraints

- V1 ships as an extension of the existing Java 17 Spring Boot 3.5 backend (modules: ad, AI, billing, template, member, pay, system, infra)
- V1 is desktop-first (1280px+), Runway-style dark UI
- AI generation via existing backend integration (GPT-4o / FLUX / etc.)
- Chinese market primary, cross-border Japan/US secondary
- Points-based consumption model (subscription + recharge) already designed in backend

## Premises

1. The wedge is unified agency workflow for multi-client, multi-channel content — not "AI generation" alone.
2. The core value is reducing integration labor from 70% to ~10% of an account manager's time.
3. V1 prioritizes the **input layer** (per-client brand templates, channel presets, product import) and **output layer** (batched export, auto file naming, platform-ready ZIP). AI generation is adequate (>80% of outputs usable without manual editing), not best-in-class. The template system (channel-matched copy structures, image specs) is an input-layer investment that directly reduces integration labor — not a generation quality play.
4. Runway dark UI differentiates from consumer-bright Chinese tools and signals "pro tool for serious work."
5. Simplicity beats completeness: one unified workflow done exceptionally well is worth more than 12 disconnected modules.
6. All 12 requirement modules are designed — but phased: V1 delivers the core agency workflow; V1.1 adds video and publishing; V1.2 adds admin backend.

## Cross-Model Perspective

An independent Claude subagent reviewed these premises and made a key challenge to premise #3 (original version focused on generation pipeline). The subagent's argument: "The AI generation is the easy middle. The pain is at the boundaries." Evidence test: watch an account manager for 2 hours and count time spent writing prompts vs. time spent renaming files, resizing images, and reformatting copy. Premise #3 was revised to prioritize input/output layers. This fundamentally reshapes V1 scope.

## Approaches Considered

### Approach A: Agency Command Center (Minimal Viable) — SELECTED
- Summary: Single-page brand dashboard → product input → AI generation → bundled ZIP export. Ships in ~1 week, proves the thesis before heavy build.
- Effort: S (human: ~1 week / CC: ~3 days)
- Risk: Low
- Pros: Fastest path to a demo that proves the 70%→10% thesis. Every feature measured against "does this reduce integration labor?"
- Cons: Limited monetization (display-only points in V1, no payment processing). Single-user. Needs rebuild for production scale.

### Approach B: Unified Agency Platform (Ideal Architecture)
- Summary: Full multi-tenant platform with auth, subscriptions, admin backend, and complete agency workflow. Production-ready from day one.
- Effort: L (human: ~3 months / CC: ~3 weeks)
- Risk: Med
- Pros: Complete monetization. Multi-tenant data isolation. Full vision realized.
- Cons: Months of build before any user touches it. High risk of building wrong thing at scale.

### Approach C: Agency RPA Layer (Creative/Lateral)
- Summary: Desktop agent sits on top of ChatGPT/Midjourney, auto-captures output, organizes by client, produces bundled exports. Zero behavior change.
- Effort: M (human: ~1 month / CC: ~1 week)
- Risk: High
- Pros: No behavior change for users. Solves actual pain without replacing tools.
- Cons: Fragile — breaks when underlying tools change UI. No moat. Doesn't build toward platform.

## Recommended Approach

**Approach A: Agency Command Center.** Ship a single-page dashboard that proves the 6x productivity thesis in one demo. Once validated with real agency account managers, evolve into Approach B (full platform). The design below covers all 12 modules from the requirements document, phased across V1, V1.1, and V1.2.

## V1 Detailed Design

### M1: Simplified Auth (V1)

V1 ships with a single login page. Registration is invite-only or manual backend account creation — no self-service signup flow in V1.

**Bootstrapping:** A seed admin account is pre-seeded in the database via SQL migration. Invite codes are one-time-use, auto-expire after 30 days, and are generated through a seed script. V1 ships with 10 invite codes. Regeneration is via a simple admin CLI command (`java -jar app.jar --generate-invites 10`). Table: `invite_codes(id, code, used, used_by, created_at, expires_at)`. The first agency account manager receives an invite code from the admin.

**Layout:**
```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│                    Wery AI                               │
│                Agency Command Center                     │
│                                                          │
│           ┌─────────────────────────────┐               │
│           │  手机号 / 邮箱              │               │
│           │                             │               │
│           │  密码                 👁    │               │
│           │                             │               │
│           │  [═══ 登录 ═══]            │               │
│           └─────────────────────────────┘               │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

**Key rules:**
- Password transmitted over HTTPS with standard form POST; backend bcrypt/Argon2 hashed storage — never plaintext. (V1 trusts HTTPS + backend hashing as the security baseline. Client-side RSA encryption adds key-distribution complexity without meaningful security gain over properly configured TLS. V1.1 re-evaluates if pen-test findings warrant additional measures.)
- Backend bcrypt/Argon2 hashed storage — never plaintext
- 3 consecutive failures → graphic captcha
- JWT + HttpOnly Cookie session management
- V1.1 adds: SMS verification login, self-service registration, invite code

### Async Job Architecture (V1)

The "Generate All" button triggers multiple concurrent AI calls (3 copy versions x N channels + 4 images x N channels = 14+ calls for 2 channels). The frontend must handle this without blocking.

**Architecture:**
1. "Generate All" creates a backend job with independent sub-jobs per channel
2. Frontend polls job status every 2 seconds
3. Results render incrementally — each channel's section populates as that channel's sub-job completes
4. The button is disabled and shows "Generating..." with a progress indicator during generation
5. Estimated total time for 2-channel generation: 2-4 minutes (GPT-4o: 5-15s/call, FLUX: 10-40s/call, with backend parallelism per channel)

**Loading UX per channel section:**
- Skeleton cards (shimmer animation) for pending results
- Cards populate with staggered fade-in as each version completes
- Progress text: "Generating Amazon JP copy... (2/3 versions)"

### Error Recovery & Reliability (V1)

**Partial failure handling:**
- Each channel's sub-job is independent. If Amazon JP copy generation fails but Amazon US succeeds, the Amazon US results display normally while Amazon JP shows an error card
- Failed card: error icon + message + "Retry" button
- Retry re-runs only the failed sub-job, not the entire generation

**Points deduction:**
- V1: Points are mock (20,000/day budget). Points display shows estimated consumption but no hard enforcement. The daily budget resets at midnight server time. If exhausted, the display shows "Daily budget used — resets at 00:00" without blocking generation
- V1.1: Points deducted only after successful completion. Failed sub-jobs do not consume points. Retried sub-jobs consume points only on success.

**Rate limits & timeouts:**
- Backend enforces per-user concurrency limit (max 5 concurrent AI calls per user, shared across all sub-jobs in the same generation request)
- Individual AI call timeout: 60 seconds
- Sub-job timeout: 3 minutes total

## V1.1 Detailed Design

### M7: Full Auth System (V1.1)

Extends M1 with:
- **Registration**: Phone + SMS verification code + password + optional invite code
- **SMS login**: Verification code as alternative to password
- **Password security**: Frontend RSA encryption → HTTPS → Backend bcrypt/Argon2
- **Brute force protection**: Progressive rate limiting → captcha → temporary lockout
- **Session**: JWT + HttpOnly Cookie, refresh token rotation

## Terminology

- **"Adequate" AI quality**: >80% of outputs usable without manual editing, validated weekly against a sample set of 50 listings per channel. "Best-in-class" would be >95%; we target 80% because the integration labor savings outweigh marginal generation quality improvements.
- **"Basic models" vs "All models"** (subscription tiers): Basic = GPT-4o-mini for copy, Stable Diffusion XL for images. All models = GPT-4o, Claude 4, FLUX, DALL-E 3, GPT Image 2. V1 uses "All models" tier for all users (mock); V1.1 enforces per subscription.
- **"Configurable" versions**: Default of 3 is a server-side constant in V1. V1.1 adds per-client setting. V1.2 adds per-job override in the UI. The "configurable" label indicates the parameter exists but the UX for changing it is deferred.
- **"Template" (copywriting/image)**: A server-side preset that maps a channel to structured AI prompt parameters. Not user-visible in V1. V1.1 adds template selection UI. V1.2 adds template editing in admin backend.
- **"Channel"**: A target e-commerce platform + locale combination (e.g., "Amazon Japan" = amazon.co.jp in Japanese). Each channel has distinct copy structure, image spec, and localization rules.

## Open Questions

1. Agency pricing model: per-seat subscription or per-client markup? Current tiers (Free/98/198) are per-user. Does the Agency tier need per-client pricing?
2. Client collaboration: do clients get view-only access to their brand workspace? V1.2 consideration.
3. Image generation model: which model (FLUX, GPT Image 2, Stable Diffusion) for which channel? Quality varies significantly for Asian vs. Western product aesthetics.
4. Multi-language quality: automated translation vs. native-language generation? Current design assumes native generation per channel, but translation fallback may be needed for smaller channels.
